## 📝 1주차 스터디 필기: 클린 코드의 기본 (정의와 명명법)
1. 클린 코드 (Clean Code)란 무엇인가?
클린 코드의 핵심은 가독성과 유지보수성입니다.

💡 정의와 중요성
클린 코드의 정의 (로버트 C. 마틴): "읽기 쉬우며, 수정하기 쉽고, 버그가 적은 코드"입니다.

코드를 읽는 데 드는 노력(인지 부하)을 최소화하여 의도가 명확히 드러나도록 작성된 코드입니다.

왜 중요한가?

유지보수 비용 절감: 소프트웨어 개발 비용의 80% 이상은 초기 개발 후의 유지보수 단계에서 발생합니다. 클린 코드는 오류를 수정하거나 기능을 추가하는 시간을 획기적으로 줄여줍니다.

협업 효율 증대: 코드는 한 번 작성되지만, 여러 사람이 수없이 읽게 됩니다. 팀원들이 코드를 빠르게 이해할 수 있어야 협업 속도가 빨라집니다.

버그 발생률 감소: 코드가 명확하고 단순할수록 숨겨진 버그가 줄어들고, 논리적 오류를 쉽게 발견할 수 있습니다.

🔍 코드를 읽는 사람
코드는 컴퓨터가 아닌 사람을 위해 작성됩니다. '나중에 올 개발자', '미래의 나 자신', 그리고 '현재의 동료'가 코드를 읽고 이해할 수 있도록 배려해야 합니다.

2. 의미 있는 이름 짓기 (Meaningful Names)
코드를 읽고 이해하는 데 있어 이름은 가장 큰 비중을 차지합니다.

1. 의도가 명확한 이름
핵심 원칙: 변수, 함수, 클래스의 이름만 보고도 무엇을 담고 있는지 또는 무엇을 하는지 분명히 알 수 있어야 합니다.

Bad: int d; (경과 시간?)

Good: int elapsedTimeInDays; (경과된 시간(일))

오해의 여지를 남기지 마세요: 실제 의미와 이름을 다르게 지어 혼란을 주지 않도록 합니다.

2. 오차를 이용한 이름 피하기
a1, a2, a3처럼 번호를 사용하거나, data, info처럼 범용적인 단어는 사용하지 마세요. 이는 이름을 짓는 노력을 회피하는 행위입니다.

3. 발음하기 쉬운 이름
이유: 코드를 논의할 때 구두로 소통하기 쉽도록 합니다.

Bad: genymdhms (generate year month day hour minute second)

Good: generationTimestamp 또는 generateDateTimeString

검색하기 쉬운 이름: 짧은 단어나 문자 하나(e.g., e)는 검색 도구로 찾기 어렵습니다. 최소한 두세 글자 이상의 의미 있는 단어를 사용하는 것이 좋습니다.

4. 명명 규칙 (Naming Convention)
클래스/객체 이름: 명사나 명사구를 사용합니다. (e.g., Customer, Account, AddressParser)

함수/메서드 이름: 동사나 동사구를 사용하며, 동사가 앞에 오도록 합니다. (e.g., saveUser, calculateTotal, deletePage)

접근자(Getter), 변경자(Setter), 조건자(Predicate)는 관례에 따라 get, set, **is**를 접두사로 사용합니다. (e.g., isAvailable, getName)

5. 맥락(Context) 추가
이름이 불완전할 경우, 해당 이름이 사용되는 맥락을 통해 의미를 보강해야 합니다.

예: firstName, lastName, street 변수들은 Address 클래스나 Customer 객체 안에 있을 때 비로소 완전한 의미를 가집니다.

3. 함수(Function)의 원칙 (단일 책임 원칙)
1. 작게 (Small)
이상적인 길이: 한 함수는 20줄을 넘지 않아야 하며, 5~10줄 내외가 가장 이상적입니다.

이유: 함수가 짧을수록 전체를 파악하기 쉽고, 버그를 찾거나 로직을 수정하기 용이해집니다.

2. 하나의 일만 (Do One Thing)
단일 책임 원칙 (Single Responsibility Principle, SRP)의 함수 버전: 함수는 오직 하나의 추상화 수준에서 하나의 작업만 수행해야 합니다.

순서 따르기: 함수 내의 모든 문장들은 같은 수준의 추상화에 머물러야 합니다. 만약 어떤 코드는 고수준의 작업(예: 데이터 처리)을 하고, 다른 코드는 저수준의 작업(예: 문자열 포맷팅)을 한다면, 이는 두 가지 일을 하고 있는 것입니다.

Bad: 데이터 유효성 검사, 데이터베이스 저장, 사용자에게 알림 전송을 하나의 함수에서 모두 처리하는 경우.

Good: createUser() 함수는 validateUserData(), saveToDatabase(), sendWelcomeEmail() 함수들을 순차적으로 호출하며, 실제 세부 구현은 하위 함수에 위임하는 경우.

3. 부수 효과 (Side Effect) 없애기
함수는 자신의 이름이 의미하는 대로만 동작해야 합니다. 함수 이름으로 예상치 못한 변경이 외부에 발생하는 것을 부수 효과라고 합니다.

Bad: checkPassword(user) 함수가 비밀번호를 확인하는 역할 외에, 사용자 객체의 세션 만료 시간(sessionExpiryTime)까지 변경하는 경우. (이는 예기치 않은 부수 효과입니다.)
